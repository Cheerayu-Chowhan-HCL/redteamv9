F03 Session Fixation — JSESSIONID Not Regenerated After Authentication HIGH
CVSS Vector
AV:N/AC:L/PR:N/UI:R/S:U/C:H/I:H/A:N
Affected Endpoint
https://demo.testfire.net/login.jsp
Evidence Summary
JSESSIONID cookie value identical before and after successful authentication. An attacker who fixes a known session token can hijack the authenticated session without credentials.
Proof of Concept
curl -s "https://demo.testfire.net/login.jsp" -X POST -d "param=[PAYLOAD]" -v
Business Impact

Exploitation of this vulnerability could result in significant data exposure or partial system compromise. Prompt remediation is strongly recommended.

Remediation

Call HttpSession.invalidate() then request.getSession(true) immediately after successful login in the authentication handler (doLogin servlet). This forces Tomcat to issue a fresh JSESSIONID bound to the new authenticated session.

F07 CSRF — No Anti-CSRF Tokens on Feedback and Login Forms MEDIUM
CVSS Vector
AV:N/AC:L/PR:N/UI:R/S:U/C:N/I:H/A:N
Affected Endpoint
https://demo.testfire.net/feedback.jsp, https://demo.testfire.net/login.jsp
Evidence Summary
Neither the feedback form (POST to /sendFeedback) nor the login form (POST to /doLogin) contain CSRF tokens. An attacker can craft a malicious page that submits these forms cross-origin on behalf of an authenticated user, enabling forged feedback submissions or login credential stuffing via CSRF.
Proof of Concept
curl -s "https://demo.testfire.net/feedback.jsp, https://demo.testfire.net/login.jsp" -X POST -d "param=[PAYLOAD]" -v
Business Impact

This vulnerability may be leveraged as part of a multi-step attack chain. While not directly exploitable for full compromise, it reduces the overall security posture.

Remediation

Implement synchroniser token pattern: generate a cryptographically random per-session CSRF token, embed it as a hidden field in every state-changing form, and validate it server-side before processing. The OWASP CSRFGuard library or Spring Security's built-in CSRF protection can be integrated with minimal code changes.

F09 Reflected XSS — search.jsp query Parameter MEDIUM
CVSS Vector
AV:N/AC:L/PR:N/UI:R/S:C/C:L/I:L/A:N
Affected Endpoint
https://demo.testfire.net/search.jsp?query=
Evidence Summary
The query parameter value is reflected directly into the HTML response body without HTML-entity encoding. A malicious script payload injected via the query parameter executes in the victim's browser context, enabling session theft, phishing, or DOM manipulation.
Proof of Concept
curl -s "https://demo.testfire.net/search.jsp?query=" -X POST -d "param=[PAYLOAD]" -v
Business Impact

This vulnerability may be leveraged as part of a multi-step attack chain. While not directly exploitable for full compromise, it reduces the overall security posture.

Remediation

HTML-encode all user-supplied output before rendering. In the search.jsp JSP, apply JSTL <c:out value="${param.query}"/> or OWASP Java Encoder: Encode.forHtml(request.getParameter("query")). Do not concatenate raw request parameters into the response body.

F10 Session Cookie Missing HttpOnly and SameSite Flags MEDIUM
CVSS Vector
AV:N/AC:L/PR:N/UI:N/S:U/C:L/I:N/A:N
Affected Endpoint
https://demo.testfire.net/
Evidence Summary
JSESSIONID cookie set without HttpOnly flag (allowing JavaScript access) and without SameSite attribute (enabling CSRF via cross-site requests). Observed in Set-Cookie header on all responses.
Proof of Concept
curl -s "https://demo.testfire.net/" -X POST -d "param=[PAYLOAD]" -v
Business Impact

This vulnerability may be leveraged as part of a multi-step attack chain. While not directly exploitable for full compromise, it reduces the overall security posture.

Remediation

Add HttpOnly and SameSite=Strict (or Lax) to the JSESSIONID cookie in Tomcat's context.xml: <SessionCookie httpOnly="true" sameSite="Strict"/>. This prevents XSS-based cookie theft and CSRF attacks targeting the session.
