Two distinct SQLi bypass payloads succeeded (length_diff -1867 and -2359 vs authenticated baseline). No WAF present. Error-based injection likely feasible given Apache/... [TRUNCATED]
Proof of Concept
curl -s "https://demo.testfire.net/doLogin" -X POST -d "param=[PAYLOAD]" -v
Business Impact

This vulnerability poses an immediate threat to the organisation. An attacker could gain full system access, exfiltrate sensitive data, or cause complete service disruption without requiring prior authentication.

Remediation

Replace raw string SQL query construction in the /doLogin authentication handler with JDBC PreparedStatement parameterised queries. Both uid and passw fields must use bound parameters. Validate/reject inputs containing SQL metacharacters as secondary control.

F08 Session Fixation — JSESSIONID Not Regenerated Post-Authentication HIGH
CVSS Vector
AV:N/AC:L/PR:N/UI:R/S:U/C:H/I:H/A:N
Affected Endpoint
https://demo.testfire.net/doLogin
Evidence Summary
test_session_fixation confirmed session_fixation_risk=true. JSESSIONID cookie identical in pre_auth_cookies and post_auth_cookies — token is not regenerated after successful authentication at /doLogin. An attacker who fixes a known session ID pre-login can hijack the authenticated session.
Proof of Concept
curl -s "https://demo.testfire.net/doLogin" -X POST -d "param=[PAYLOAD]" -v
Business Impact

Exploitation of this vulnerability could result in significant data exposure or partial system compromise. Prompt remediation is strongly recommended.

Remediation

Call HttpSession.invalidate() and issue a new session token immediately upon successful authentication at /doLogin. The JSESSIONID must change on privilege escalation — configure the application server (Apache Tomcat) to regenerate session IDs on login.
